Q.1(a) Do as Directed — Definitions & Short Answers 7 Marks
1) Define Database.
Database: A database is an organized collection of related data stored in a structured format so that it can be easily accessed, managed, and updated. It represents some aspect of the real world (called a miniworld or Universe of Discourse).

Example: A university database stores data about students, courses, faculty, and grades.
2) Define Entity.
Entity: An entity is a real-world object or thing that has a distinct existence and about which data is stored in a database. It can be a person, place, object, event, or concept.

Example: In a university database, Student, Course, and Faculty are entities.
3) Define Primary Key.
Primary Key: A primary key is an attribute (or a set of attributes) that uniquely identifies each tuple (row) in a relation (table). It must satisfy two properties:
  • Uniqueness — No two rows can have the same primary key value.
  • Not Null — Primary key cannot have a NULL value.
Example: In a STUDENT table, Roll_No can be the primary key.
4) BCNF stands for __________.
BCNF stands for: Boyce-Codd Normal Form

It is a stricter version of 3NF. A relation is in BCNF if for every functional dependency X → Y, X must be a super key of the relation.
5) Full Form of DML.
DML = Data Manipulation Language

DML consists of SQL commands used to manipulate data stored in a database. Commands include: SELECT, INSERT, UPDATE, DELETE.
6) Full Form of ERD.
ERD = Entity-Relationship Diagram

An ERD is a graphical representation that shows the entities, their attributes, and the relationships between them. It is used in the conceptual design phase of a database.

Q.1(b) What is DBMS? Explain Advantages and Disadvantages 7 Marks
DBMS (Database Management System): A DBMS is a software system that allows users to define, create, maintain, and control access to a database. It acts as an interface between the user and the database.
Examples: MySQL, Oracle, MS SQL Server, PostgreSQL.

Advantages of DBMS

  1. Data Sharing: Multiple users can access the same data simultaneously in a controlled manner.
  2. Reduced Data Redundancy: Data is stored at one place, avoiding duplication (unlike file systems).
  3. Data Integrity: Integrity constraints (Primary Key, Foreign Key) ensure accuracy and consistency of data.
  4. Data Security: DBMS provides access control — only authorized users can access specific data.
  5. Data Abstraction: Users interact with data without knowing how it is physically stored (Three-Level Architecture).
  6. Backup and Recovery: DBMS provides mechanisms to recover data in case of failure.
  7. Concurrent Access: Multiple users can access data at the same time without conflicts (through locking/concurrency control).

Disadvantages of DBMS

  1. High Cost: DBMS software is expensive (commercial ones like Oracle). Hardware requirements are also costly.
  2. Complexity: Setting up and managing a DBMS requires skilled database administrators (DBA).
  3. Large Size: DBMS software is very large and requires significant disk space and memory.
  4. Performance Overhead: For simple applications, DBMS may be slower than file-based systems due to extra overhead.
  5. Single Point of Failure: If the DBMS crashes, all applications using it are affected.
Write at least 4 advantages and 3 disadvantages with one-line explanations each to score full marks.
Q.2(a) Classifications of DBMS — Explain any Two 7 Marks

Classifications of DBMS

DBMS can be classified based on the data model used:

  1. Hierarchical DBMS
  2. Network DBMS
  3. Relational DBMS (RDBMS)
  4. Object-Oriented DBMS
  5. NoSQL DBMS

1. Hierarchical DBMS

Definition: Data is organized in a tree-like (parent-child) structure. Each parent can have multiple children, but each child has only one parent.

Example: IBM's IMS (Information Management System).
ROOT / \ Dept A Dept B / \ | Emp1 Emp2 Emp3
  • Advantage: Fast data access for predefined relationships.
  • Disadvantage: Rigid structure; does not handle many-to-many relationships well.

2. Relational DBMS (RDBMS)

Definition: Data is stored in tables (relations) consisting of rows (tuples) and columns (attributes). Tables are linked using keys.

Based on: E.F. Codd's relational model (1970).
Examples: MySQL, Oracle, MS SQL Server, PostgreSQL.
Example — STUDENT Table
Roll_No | Name | City
--------|---------|------
101 | Ravi | Surat
102 | Priya | Ahmedabad
  • Advantage: Simple structure, easy querying with SQL, supports data integrity.
  • Disadvantage: May be slow for very large datasets compared to NoSQL.
You only need to explain any TWO in detail. Hierarchical + Relational is the safest choice as both are well-understood.

Q.3(a) What is Normalization? Explain 1NF, 2NF and 3NF 7 Marks
Normalization: Normalization is the process of organizing data in a database to reduce data redundancy and improve data integrity. It involves decomposing a large table into smaller, well-structured tables without losing data.

1NF — First Normal Form

Rule: A relation is in 1NF if:

  • All attributes contain atomic (indivisible) values.
  • Each column has a unique name.
  • Each row is unique (has a primary key).
  • No repeating groups or multi-valued attributes.
Example — Violation (NOT in 1NF)
StudentID | Name | Phone
101 | Ravi | 9876, 1234 ← Multiple values!
Converted to 1NF
StudentID | Name | Phone
101 | Ravi | 9876
101 | Ravi | 1234

2NF — Second Normal Form

Rule: A relation is in 2NF if:

  • It is already in 1NF.
  • There is no Partial Dependency — every non-key attribute must be fully functionally dependent on the entire primary key (not just a part of it).
  • Applies only when the primary key is composite.
Example — Violation (NOT in 2NF)
Table: ENROLL(SID, CID, MARKS, STUDENT_NAME)
Primary Key = (SID, CID)
STUDENT_NAME depends only on SID → Partial Dependency!
Converted to 2NF (split tables)
ENROLL(SID, CID, MARKS) — PK = (SID, CID)
STUDENT(SID, STUDENT_NAME) — PK = SID

3NF — Third Normal Form

Rule: A relation is in 3NF if:

  • It is already in 2NF.
  • There is no Transitive Dependency — no non-key attribute depends on another non-key attribute.
Example — Violation (NOT in 3NF)
STUDENT(SID, Name, Zip, City)
SID → Zip → City — Zip is non-key, City depends on Zip → Transitive!
Converted to 3NF
STUDENT(SID, Name, Zip)
LOCATION(Zip, City)
Draw a small table example for each NF. This is what examiners look for — not just definitions.
Q.3(b) Define Join. List and Explain Types of Join 7 Marks
Join: A Join is a binary relational algebra operation that combines tuples from two relations based on a related attribute (condition). It is used to retrieve data from multiple tables.

Types of Join

1. Inner Join (Theta Join / Equi Join / Natural Join)

Theta Join: Combines tuples where a condition (θ) is satisfied. Condition can be =, >, <, ≠.

R ⋈θ S = σθ(R × S)

Equi Join: Theta Join where the condition uses only equality (=).

Natural Join (⋈): Equi Join that automatically matches attributes with the same name and removes duplicate columns.

Example — Natural Join
STUDENT(SID, Name) ⋈ ENROLL(SID, CID, Marks)
Result: SID | Name | CID | Marks

2. Outer Join

Outer Join includes tuples that do not have matching counterparts in the other relation, filling unmatched attributes with NULL.

  • Left Outer Join (⟕): All tuples from the LEFT relation + matched tuples from right. Unmatched right values = NULL.
  • Right Outer Join (⟖): All tuples from the RIGHT relation + matched tuples from left. Unmatched left values = NULL.
  • Full Outer Join (⟗): All tuples from BOTH relations. Unmatched values on either side = NULL.
Join TypeReturnsSymbol
Inner / Natural JoinOnly matching tuples⋈
Left Outer JoinAll left + matched right⟕
Right Outer JoinAll right + matched left⟖
Full Outer JoinAll tuples from both⟗

3. Self Join

A table is joined with itself. Used when a relation has a recursive relationship.

Example
EMPLOYEE(EmpID, Name, ManagerID)
Self Join to find Employee and their Manager Name.
Always write the join symbol and a one-line example for each type. Left/Right Outer Join with NULL example impresses examiners.

Q.3(b) OR Multivalued Dependency and 4NF 7 Marks
Multivalued Dependency (MVD): A multivalued dependency X →→ Y exists in a relation R when, for each value of X, there is a set of values of Y that is independent of all other attributes Z in R. In simple terms: X determines a set of values of Y.

Example of MVD

Relation: EMPLOYEE(EmpID, Skill, Hobby)
EmpID | Skill | Hobby
101 | Java | Cricket
101 | Java | Chess
101 | Python | Cricket
101 | Python | Chess

Here: EmpID →→ Skill and EmpID →→ Hobby independently.
This causes redundancy — MVD problem!

4NF — Fourth Normal Form

Rule: A relation is in 4NF if:
  • It is in BCNF, AND
  • It has no non-trivial Multivalued Dependencies, except when the left side is a superkey.

Converting to 4NF

Decompose the relation to eliminate MVDs:

After 4NF Decomposition
EMP_SKILL(EmpID, Skill)
EMP_HOBBY(EmpID, Hobby)

Now each table handles one MVD, eliminating redundancy.

Define MVD with the double arrow (→→) notation. Show before/after tables. This scores full marks.
Q.4(a) Union, Intersection and Difference (Minus) Operations 7 Marks

These are Set Theory Operations in Relational Algebra. They require both relations to be union-compatible (same degree and compatible domains).

Prerequisite: Union Compatibility

Two relations R and S are union-compatible if: they have the same number of attributes AND corresponding attributes have the same domain.

1. UNION (∪)

R ∪ S: Returns all tuples that are in R or S or both. Duplicate tuples are eliminated.
Example
R (Students in Java): SID: {101, 102, 103}
S (Students in Python): SID: {102, 103, 104}

R ∪ S = {101, 102, 103, 104} ← All unique students

2. INTERSECTION (∩)

R ∩ S: Returns only the tuples that appear in both R and S.
Example
R ∩ S = {102, 103} ← Students enrolled in BOTH Java AND Python

3. DIFFERENCE / MINUS (−)

R − S: Returns tuples that are in R but NOT in S.
Example
R − S = {101} ← Students in Java but NOT in Python
S − R = {104} ← Students in Python but NOT in Java

Summary Table

OperationSymbolReturns
Union∪Tuples in R OR S (no duplicates)
Intersection∩Tuples in BOTH R AND S
Difference−Tuples in R but NOT in S

  • HAVING: Filters groups after aggregation (similar to WHERE but for groups).

Q.4(a) OR Entity Integrity, Referential Integrity and Foreign Keys 7 Marks

1. Entity Integrity

Entity Integrity Constraint: States that the Primary Key of a relation cannot be NULL and must be unique. Every tuple must be uniquely identifiable.

Rule: For every tuple t in relation R, the value of the primary key PK cannot be null.

Example: In STUDENT(Roll_No, Name, City), Roll_No is the primary key. No student record can have a NULL or duplicate Roll_No.

Why? Without entity integrity, we cannot uniquely identify rows, making data management impossible.

2. Referential Integrity

Referential Integrity Constraint: States that a Foreign Key value in one relation must either be NULL or must match a value of the Primary Key in the referenced relation.

Rule: If a tuple in relation R has a foreign key FK, then FK must match the primary key of some tuple in relation S, OR be NULL.
Example
STUDENT(SID, Name) ← Primary Key: SID
ENROLL(SID, CID, Marks) ← Foreign Key: SID references STUDENT(SID)

→ You cannot insert a row in ENROLL with SID = 999 if SID 999 does not exist in STUDENT.

3. Foreign Key

Foreign Key: A Foreign Key is an attribute (or set of attributes) in one relation that references the Primary Key of another relation. It establishes a link between two tables.

Key Points:
  • Foreign Key can have NULL values (unless restricted).
  • Foreign Key can have duplicate values.
  • It maintains referential integrity.

Referential Actions (on delete/update)

ActionDescription
CASCADEAutomatically deletes/updates child rows when parent is deleted/updated
SET NULLSets the foreign key to NULL when parent is deleted
RESTRICTPrevents deletion of parent if child rows exist
NO ACTIONSimilar to RESTRICT — default behavior

Q.4(b) OR SELECTION and PROJECTION — Unary Relational Operations 7 Marks

Both SELECT and PROJECT are Unary Operations in Relational Algebra — they operate on a single relation.

1. SELECT Operation (σ)

SELECT (σ): The select operation retrieves only those tuples (rows) from a relation that satisfy a given condition (predicate). It is a horizontal filter — it reduces rows.
σ <condition> (R)
Example — STUDENT(SID, Name, City, Marks)
σ City = 'Surat' (STUDENT)
→ Returns all students from Surat.

σ Marks > 70 (STUDENT)
→ Returns all students who scored more than 70 marks.

σ (City = 'Surat') ∧ (Marks > 60) (STUDENT)
→ Students from Surat with marks above 60.

Properties of SELECT:

  • The degree (number of columns) of the result is same as original relation.
  • Number of tuples in result ≤ number of tuples in R.
  • SELECT is commutative: σc1(σc2(R)) = σc2(σc1(R))

2. PROJECT Operation (π)

PROJECT (π): The project operation selects only certain attributes (columns) from a relation, removing others. It is a vertical filter — it reduces columns. Duplicate tuples are automatically removed.
π <attribute list> (R)
Example — STUDENT(SID, Name, City, Marks)
π Name (STUDENT)
→ Returns only the Name column of all students.

π Name, City (STUDENT)
→ Returns only Name and City columns.

Properties of PROJECT:

  • Number of tuples in result ≤ tuples in R (duplicates removed).
  • PROJECT is NOT commutative.
  • Degree of result = number of attributes in the attribute list.

Combining SELECT and PROJECT

Get names of students from Surat
π Name (σ City='Surat' (STUDENT))
SELECT (σ)PROJECT (π)

Q.5(a) ACID Properties of Transaction with Example 7 Marks
Transaction: A transaction is a logical unit of work that contains one or more database operations (read/write). All operations in a transaction must complete successfully, or none of them should take effect.

ACID Properties

ACID stands for: Atomicity, Consistency, Isolation, Durability.

1. Atomicity

A transaction is treated as a single unit — either all operations are executed successfully (COMMIT) or none of them are (ROLLBACK). There is no partial execution.
Example — Bank Transfer: A sends ₹500 to B
T1: Debit A's account by ₹500
T2: Credit B's account by ₹500

If T1 succeeds but system crashes before T2 → both are rolled back. Money is not lost.

2. Consistency

A transaction must bring the database from one consistent state to another consistent state. All integrity constraints must be satisfied before and after the transaction.
Example
Total money in A + B = ₹1000 (before transfer).
After transfer of ₹500: A has ₹500 less, B has ₹500 more → total still ₹1000. Consistent!

3. Isolation

Transactions execute in isolation — the intermediate state of a transaction is not visible to other concurrent transactions. Each transaction appears to execute as if it were the only one running.
Example
If T1 is transferring money and T2 is checking balance simultaneously,
T2 should not see the intermediate (half-done) state of T1.

4. Durability

Once a transaction is committed, its changes are permanently saved in the database, even if there is a system failure (power outage, crash). This is ensured by transaction logs and checkpoints.
Example
After the bank transfer is committed and confirmed, even if the server crashes,
the transfer remains in the database when it restarts.
PropertyGuaranteesMechanism
AtomicityAll or nothingCOMMIT / ROLLBACK
ConsistencyValid state before & afterIntegrity constraints
IsolationNo interference between transactionsConcurrency control (locking)
DurabilityCommitted data survives failuresWrite-ahead logging, checkpoints
Use the Bank Transfer example for ALL four properties — it's a classic GTU-friendly example that covers all four clearly.
Q.5(b) Define Transaction. Draw and Explain Transaction State Diagram 7 Marks
Transaction: A transaction is a sequence of one or more database operations (READ, WRITE) that together form a single logical unit of work. A transaction must satisfy the ACID properties.

Transaction State Diagram

┌─────────┐ Begin ┌─────────┐ Partially ┌─────────────┐ │ ACTIVE │ ──────────── ─│ ACTIVE │── Committed ──▶│ PARTIALLY │ └─────────┘ └─────────┘ │ COMMITTED │ │ │ └─────────────┘ │ Failure │ Failure │ ▼ ▼ Success │ Failure ┌─────────┐ ┌──────────┐ ▼ ▼ │ FAILED │──Rollback──▶│ ABORTED │ ┌───────────┐ ┌──────────┐ └─────────┘ └──────────┘ │ COMMITTED │ │ FAILED │ └───────────┘ └──────────┘

States Explained

StateDescription
ActiveThe initial state. Transaction is being executed (read/write operations are happening).
Partially CommittedThe final operation of the transaction has been executed but not yet permanently saved. Changes are in memory buffer.
CommittedTransaction has successfully completed. Changes are permanently written to the database. Cannot be rolled back.
FailedTransaction cannot proceed due to an error (hardware/software failure, constraint violation). Must be rolled back.
Aborted (Rolled Back)Transaction has been rolled back and the database is restored to its state before the transaction began.

State Transitions

  • Active → Partially Committed: When the last operation of the transaction is executed.
  • Partially Committed → Committed: When all changes are successfully written to disk.
  • Partially Committed → Failed: When a failure occurs just before committing.
  • Active → Failed: When an error occurs during execution.
  • Failed → Aborted: After rollback is complete, database is restored.

Q.5(a) OR Characterizing Schedules Based on Serializability 7 Marks
Schedule: A schedule is an ordering of the operations (Read/Write) from multiple concurrent transactions. A good schedule should produce the same result as some serial (one at a time) execution of those transactions.
Serial Schedule: Transactions execute one after another with no interleaving. Always correct but has poor performance (no concurrency).
Example: T1 completes fully → then T2 starts.
Serializable Schedule: A concurrent schedule is serializable if its result is equivalent to some serial schedule. This ensures correctness while allowing concurrency.

Types of Serializability

1. Conflict Serializability

Two operations conflict if:

  • They belong to different transactions.
  • They access the same data item.
  • At least one of them is a WRITE operation.

A schedule is conflict serializable if it can be converted to a serial schedule by swapping non-conflicting operations.

Example
T1: R(A), W(A) T2: R(A), W(A)

Schedule S: R1(A), R2(A), W1(A), W2(A)
W1(A) and R2(A) conflict → Check Precedence Graph

Precedence Graph (Serialization Graph): Draw a directed graph where an edge Ti → Tj means Ti must come before Tj. If the graph has NO cycle, the schedule is conflict serializable.

2. View Serializability

A schedule S is view serializable if it is view equivalent to some serial schedule. It is a weaker (broader) condition than conflict serializability. Every conflict serializable schedule is view serializable, but not vice versa.

Types of Schedules Summary

Schedule TypeDescription
SerialNo interleaving; always correct
Non-serialOperations interleaved; may or may not be correct
SerializableEquivalent to a serial schedule; correct
Non-serializableNot equivalent to any serial schedule; incorrect
Drawing the precedence graph and checking for cycles is the standard method to test conflict serializability — always include it.
